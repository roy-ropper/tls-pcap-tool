# tls-pcap-tool

Decrypt your own client's TLS traffic in Wireshark, for traffic generated by
Firefox, curl, and Python (`requests`, `urllib3`, `urllib`) on Debian/Ubuntu.

This is **not** a MITM proxy and does not strip or intercept TLS. It uses the
standard `SSLKEYLOGFILE` mechanism: TLS clients log their per-session secret
keys to a file as they negotiate connections, and Wireshark reads that file to
decrypt the *same* ciphertext it captures on the wire. No certificate
substitution, no interception — this only works because you own the client
generating the traffic.

## What the script does

Running `tls-debug.sh` will, every time (idempotent, safe to re-run):

1. Create `~/tls-debug/sslkeylog.log` (mode 600 — it holds secret key
   material).
2. Ensure `export SSLKEYLOGFILE="$HOME/tls-debug/sslkeylog.log"` is present
   and correct in `~/.bashrc`, normalizing/fixing any stale or differently
   formatted prior entry.
3. Configure Wireshark's `tls.keylog_file` preference
   (`~/.config/wireshark/preferences`) to point at the same file, so even a
   manually-launched Wireshark picks it up.
4. If run with `sudo`, launch Wireshark as root (so `dumpcap` can list and
   capture on interfaces without needing the `wireshark` group) and leave you
   to pick the interface and start the capture yourself — no interface name
   is hardcoded, since it varies machine to machine.

curl and modern Python (`ssl` module 3.8+, and `urllib3` as used by
`requests`) already read `SSLKEYLOGFILE` natively — no extra shim or patching
is installed for them, since that was tested and found not just unnecessary
but actively harmful on a previous iteration of this script (see *Notes*
below).

Firefox reads `SSLKEYLOGFILE` natively too, but only at process startup — if
Firefox is already running, fully quit it before relying on the new value.

## Usage

```bash
sudo ./tls-debug.sh
```

Then:

1. In the Wireshark window that opens, pick your capture interface and start
   capturing.
2. Open a **new** terminal (so it picks up `~/.bashrc`), or run
   `source ~/.bashrc` in your current one.
3. Start Firefox (a fresh process, not a reused one) and browse to any HTTPS
   site, or run curl/Python from the terminal you just sourced:
   ```bash
   curl -s https://example.com -o /dev/null
   python3 -c "import requests; requests.get('https://example.com')"
   ```
4. In Wireshark, filter on `tls`, then right-click a TLS Application Data
   packet → Follow → HTTP/HTTP2 Stream to see the decrypted plaintext.
5. Sanity check: `tail -f ~/tls-debug/sslkeylog.log` should grow as you
   generate traffic.

If you run the script without `sudo`, it still does all the setup but skips
launching Wireshark (it prints a reminder to re-run with `sudo` when you're
ready to capture) — privileged capture access is intentionally left to you to
invoke explicitly, on each machine, rather than this script modifying group
membership or system accounts.

## Requirements

- Debian/Ubuntu-based system (uses `~/.bashrc`, `apt`-installed `wireshark`)
- `wireshark`, `curl`, `python3`, and `firefox` already installed
- An X11 session (the script forwards `DISPLAY`/`XAUTHORITY` so Wireshark can
  open a window when launched via `sudo`)

## Notes / known gotchas

- **No group/account changes are made.** You invoke `sudo` yourself per
  machine; the script never runs `usermod` or touches system accounts.
- **Snap-packaged Firefox is AppArmor-confined.** Its `home` interface denies
  writes to hidden (dot) directories — this is why the keylog path is
  `~/tls-debug/`, not `~/.tls-debug/`. Don't change it back to a hidden
  directory; Firefox will silently fail to write to it.
- **`.bashrc` only runs for interactive shells.** Non-interactive scripts,
  cron jobs, etc. won't see `SSLKEYLOGFILE` from it — only real terminals you
  open will.
- **Do not add a Python `usercustomize.py`/`sitecustomize.py` shim to patch
  `ssl.SSLContext` for this.** An earlier version of this tool did exactly
  that to force `keylog_filename`, and it broke every single Python TLS
  connection on the system: patching `ssl.SSLContext.__init__` defeats a
  CPython special case (a class that overrides `__new__` but not `__init__`
  skips the "too many arguments" check; patching `__init__` re-enables that
  check and `SSLContext()` starts raising `TypeError`). It also turned out to
  be unnecessary — verified by reading `ssl.py` and `urllib3/util/ssl_.py` —
  both already read `SSLKEYLOGFILE` on their own.
